# 安卓 OTA 升级异常注入与多机验证平台项目规划书

## 摘要
- 项目名称定为 `AegisOTA`，定位是“面向测试开发岗位的安卓系统升级异常注入与多机验证平台”。
- 项目核心叙事不是做一个普通测试后台，而是把你的真实经验产品化：围绕 `OTA/补丁升级/回滚/升级后稳定性验证` 建立可编排、可追踪、可复盘的平台。
- 主卖点锁定为两条主线：第一条是“升级异常注入与状态机验证”，第二条是“多设备机房编排与资源隔离”。
- 技术栈锁定为纯 Python：`FastAPI + SQLAlchemy + SQLite + Typer + Jinja2/HTMX`。这样项目从接口层、调度层到展示层都能保持 Python 主导，和你的背景一致。
- 最终交付物不是单一脚本，而是一套可讲清楚架构的完整作品：轻 Web 控制台、CLI 执行器、设备编排、升级任务状态机、报告系统、日志归因与异常复盘链路。

## 关键设计
### 1. 产品目标与面试叙事
- 对外介绍时，项目解决的问题是：真实安卓升级测试依赖大量人工判断、机房资源冲突严重、异常复现困难、升级后的回归链路割裂。
- 平台目标是把“升级前检查、升级执行、异常注入、升级后验证、结果归因、设备回收”串成一条完整流水线。
- 面试讲解重点放在三个技术层面：升级状态机设计、设备调度与并发控制、异常归因与质量门禁。
- 简历项目标题建议写成：`安卓 OTA 升级异常注入与多机验证平台（Python）`。
- 简历项目描述建议围绕“平台化、编排、状态建模、异常注入、报告归因”这些词展开，而不是“写了很多脚本”。

### 2. 功能范围与边界
- 必做主线一：升级流程编排。支持 `全量升级`、`增量 patch`、`失败回滚`、`升级后验证` 四类任务模板。
- 必做主线二：异常注入。支持 `电量不足`、`存储压力`、`下载中断`、`校验失败`、`重启打断`、`升级后 watchdog/启动异常判定`、`升级后 monkey 压测`。
- 必做主线三：多机房设备管理。支持设备注册、标签分组、占用租约、健康检查、故障隔离、任务并发上限、紧急停止。
- 必做主线四：结果归因与报告。输出任务时间线、关键日志片段、失败分类、设备维度统计、场景维度通过率。
- 不做内容明确排除：不实现 Android 系统内核级改造，不做真正的 COW 内核逻辑开发，不做复杂前后端分离，不做云原生部署，不做大而全权限系统。
- COW、watchdog、patch 应作为“升级场景与异常模型”的一部分进行建模和日志关联，不承诺实现底层源码级改写；这是面试里必须明确的边界，否则会显得不可信。

### 3. 系统架构与接口
- 架构采用“控制面 + 执行面”。
- 控制面是 `FastAPI` 服务，负责设备清单、任务定义、调度、报告和 Web 展示。
- 执行面是 `Typer` CLI 和后台 worker，负责调用 `adb/fastboot/cmd`、采集日志、执行 monkey、跑性能检查、上报事件。
- Web 层用 `Jinja2 + HTMX`，页面只做任务看板、设备看板、任务详情、日志与报告查看，避免引入 JS 工程复杂度。
- 数据层用 `SQLite + SQLAlchemy`，核心实体固定为：`Device`、`DeviceLease`、`UpgradePlan`、`FaultProfile`、`RunSession`、`RunStep`、`Artifact`、`Report`。
- 调度采用单机可控方案：数据库状态机 + Python worker 循环，不引入 Redis/Celery。任务状态固定为 `queued -> reserved -> running -> validating -> passed/failed/aborted/quarantined`。
- 设备并发策略固定为“单设备同一时刻只允许一个独占任务，多设备任务按标签池和并发额度分配，异常设备自动进入 quarantine”。
- 升级状态机固定为五段：`precheck`、`push_package`、`apply_update`、`reboot_wait`、`post_validate`。每段都允许注入 fault hook。
- 异常注入接口固定为三个插件点：`PreUpgradeFault`、`ApplyStageFault`、`PostBootFault`。实现方式以外部命令、环境条件模拟、日志检测为主。
- 日志采集固定包含：`logcat`、关键 shell 输出、任务事件时间线、monkey 摘要、性能摘要、失败截图或文本证据。
- Web API 固定提供以下接口：
- `POST /api/runs` 创建升级任务。
- `GET /api/runs/{id}` 查询任务详情和阶段状态。
- `POST /api/runs/{id}/abort` 终止任务。
- `GET /api/devices` 查询设备状态、标签、租约和健康度。
- `POST /api/devices/{id}/quarantine` 隔离异常设备。
- `GET /api/reports/{id}` 返回报告摘要与证据链。
- CLI 固定提供以下命令：
- `labctl device sync` 扫描并更新在线设备。
- `labctl run submit` 提交升级任务。
- `labctl run execute` 由 worker 执行具体设备任务。
- `labctl report export` 导出 Markdown/HTML 报告。
- `labctl device recover` 处理失败设备回收与重新入池。

### 4. 实现计划
- 第 1 周：完成需求定型、数据模型、任务状态机、设备模型、CLI 骨架、FastAPI 基础接口和 SQLite 持久化。
- 第 2 周：完成真实设备执行链路，至少打通 `设备发现 -> package 下发 -> 升级命令执行 -> 重启等待 -> logcat 抓取 -> 结果入库`。
- 第 3 周：完成异常注入框架和首批 fault profile，至少落地 `下载中断`、`存储压力`、`重启打断`、`升级后 monkey` 四类场景。
- 第 4 周：完成多设备编排、租约机制、隔离机制、轻 Web 页面、任务时间线和失败归因报告。
- 第 5 周：补齐报告导出、示例数据、演示脚本、架构图、README、简历描述和面试讲稿。
- 仓库内容同时准备三类材料：可运行代码、设计文档、面试讲解材料。面试时不演示真实设备也能完整讲清楚方案。

## 测试与验收
- 单元测试必须覆盖升级状态机转换、fault profile 参数校验、失败分类逻辑、设备租约竞争、任务终止逻辑。
- 集成测试必须通过 mock `adb/fastboot/subprocess` 跑通完整任务，验证数据库状态流转和报告输出。
- 端到端验收至少包含四个场景：正常全量升级、增量 patch 失败后回滚、升级后 monkey 触发异常、设备中途离线进入 quarantine。
- 报告验收标准固定为：每次任务都能生成任务时间线、设备信息、关键日志片段、失败结论、下一步建议。
- 平台验收标准固定为：本地单机可部署；支持至少 10 台设备的资源建模；同一设备不存在并发任务冲突；异常设备可隔离并恢复；任一任务可追溯到执行命令与日志证据。
- 面试验收标准固定为：可以在 5 分钟内讲清楚问题背景、架构拆分、核心状态机、异常注入设计、为什么这样取舍。

## 默认假设
- 项目以真实设备导向设计，但不要求面试现场演示；因此必须保留 mock 执行器与离线回放能力，用于测试和讲解。
- 设备环境默认是你可访问的 Android 测试机，不依赖 root 才能完成主流程；对 root 或工程机能力的场景只作为可选扩展，不作为主线。
- COW、watchdog、patch 等底层概念在项目中体现为“升级阶段建模 + 故障场景抽象 + 日志归因规则”，不做源码级内核实验。
- 前端不作为卖点，展示层只做“足够像产品”的轻量界面，不投入前后端分离成本。
- 数据库默认单机 SQLite，不追求高可用部署；如果后续你想往 Python 后端岗位再偏一点，再演进到 Postgres 双模式。
- 项目成功标准不是“功能越多越好”，而是“有一条足够硬的主线能讲透，并且第二条主线能证明平台思维”。
